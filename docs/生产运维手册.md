# AeroLink 生产运维手册

## 基本信息

- 服务器：`101.43.50.30`
- 系统用户：`ubuntu`
- 应用目录：`/home/ubuntu/aerolink-prod`
- 当前云端 Compose 文件：`docker-compose.prod.sqlite.yml`
- 目标 PostgreSQL Compose 文件：`docker-compose.prod.yml`
- 环境变量文件：`.env.production`
- 公网入口：`http://101.43.50.30:8080`
- 健康检查：`http://101.43.50.30:8080/api/health`

> 上述 HTTP 地址仅保留作历史运行态记录。批次二默认启用 `REQUIRE_SECURE_ORIGIN=true`，下次生产启动前必须切换到域名 HTTPS（TLS profile 或受信任的上游 TLS 终止），不要把该 HTTP 地址作为正式登录入口。

建议先在服务器上进入应用目录：

```bash
cd /home/ubuntu/aerolink-prod
export COMPOSE_FILE=docker-compose.prod.sqlite.yml
```

说明：

- 当前 `101.43.50.30` 仍运行在 SQLite 兼容 profile，上面的 `COMPOSE_FILE` 可直接用于现网发布。
- 等正式迁移到 PostgreSQL 后，把 `COMPOSE_FILE` 改成 `docker-compose.prod.yml` 即可。
- 新 PostgreSQL 环境默认走 `SCHEMA_MIGRATION_MODE=migrate`，只有在备份/演练环境完成迁移基线后才可设置 `MIGRATION_BASELINE_CONFIRMED=true`。
- 生产入口默认关闭演示 seed，并要求存在至少一个已激活的管理员/总经理账号。
- 生产 refresh cookie 为 Secure，`REQUIRE_SECURE_ORIGIN=true` 时 `CLIENT_URL` 必须是 HTTPS；HTTP 兼容入口只能放在 TLS profile 或上游 HTTPS 代理之后。

## 重启与启停

启动或拉起现有服务：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" up -d
```

停止服务：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" down
```

重启整个栈：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" down
docker compose --env-file .env.production -f "$COMPOSE_FILE" up -d
```

只重启前端或后端：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" restart web
docker compose --env-file .env.production -f "$COMPOSE_FILE" restart backend
```

## HTTPS profile 与发布验收

默认 Compose 保留 8080 兼容入口；正式域名应准备受信任证书目录（包含 `fullchain.pem`、`privkey.pem`），再叠加 TLS profile：

```bash
export TLS_CERT_DIR=/etc/letsencrypt/live/aerolink.example.com
docker compose --env-file .env.production \
  -f docker-compose.prod.yml -f docker-compose.prod.tls.yml up -d --build
```

TLS profile 只绑定 80/443，80 会重定向到 HTTPS，并在 HTTPS 响应中校验 HSTS、CSP、X-Content-Type-Options、X-Frame-Options、Referrer-Policy 和 Permissions-Policy。切流后在服务器上运行：

```bash
BASE_URL=https://aerolink.example.com bash deploy/verify-production.sh
```

脚本只读检查 `/api/health` 和安全响应头，不执行迁移、seed 或数据写入。

## 首次管理员初始化

在新环境或恢复后没有活动特权账号时，先通过一次性命令创建未激活管理员。命令不会生成或输出默认密码，只输出一次性激活链接：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" run --rm --no-deps \
  -e BOOTSTRAP_ADMIN_EMAIL=<管理员邮箱> \
  -e BOOTSTRAP_ADMIN_NAME="平台管理员" \
  --entrypoint sh backend -lc 'npm run db:bootstrap-admin'
```

管理员在激活页面自行设置密码后，再启动或重启 backend。若没有活动特权账号，生产入口的 `db:check-active-admin` 门禁会拒绝启动。

演示数据只允许在隔离环境使用，并且必须显式设置 `DEMO_SEED_PASSWORD`；生产环境还需额外设置 `ALLOW_PRODUCTION_DEMO_SEED=true`，不应把该开关作为常规发布步骤。

## 发布更新

### 1. 从本机同步变更到服务器

Windows 环境建议显式使用系统 `ssh.exe` 和 `scp.exe`，避免 PATH 或 Git 自带 SSH 干扰。

```powershell
$Key = "<pem 文件路径>"
& "C:\Windows\System32\OpenSSH\scp.exe" -i $Key -o StrictHostKeyChecking=accept-new .\package.json ubuntu@101.43.50.30:/home/ubuntu/aerolink-prod/package.json
& "C:\Windows\System32\OpenSSH\scp.exe" -i $Key -o StrictHostKeyChecking=accept-new .\package-lock.json ubuntu@101.43.50.30:/home/ubuntu/aerolink-prod/package-lock.json
```

如果有更多文件变更，按同样方式同步到 `/home/ubuntu/aerolink-prod` 对应路径。

### 2. 服务器上执行更新

前端变更：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" build web
docker compose --env-file .env.production -f "$COMPOSE_FILE" up -d web
```

后端变更：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" build backend
docker compose --env-file .env.production -f "$COMPOSE_FILE" up -d backend
```

全量更新：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" up -d --build
```

更新后建议立即验证：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" ps
wget -qO- http://127.0.0.1:8080/api/health
```

## 数据备份

当前云端 SQLite 兼容 profile 使用两个 Docker volume：

- `aerolink-prod_backend-data`：SQLite 数据库
- `aerolink-prod_backend-uploads`：上传文件

如果已经切换到 PostgreSQL profile，请改为备份 `aerolink-prod_postgres-data`，并优先使用 `pg_dump` 做逻辑备份。

先准备备份目录：

```bash
mkdir -p /home/ubuntu/backups
```

备份数据库卷：

```bash
docker run --rm -v aerolink-prod_backend-data:/from -v /home/ubuntu/backups:/to alpine sh -lc 'tar czf /to/backend-data-$(date +%F-%H%M%S).tgz -C /from .'
```

备份上传文件卷：

```bash
docker run --rm -v aerolink-prod_backend-uploads:/from -v /home/ubuntu/backups:/to alpine sh -lc 'tar czf /to/backend-uploads-$(date +%F-%H%M%S).tgz -C /from .'
```

查看已有备份：

```bash
ls -lh /home/ubuntu/backups
```

## 数据恢复

恢复前先停服务：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" down
```

恢复数据库卷：

```bash
docker run --rm -v aerolink-prod_backend-data:/target -v /home/ubuntu/backups:/backup alpine sh -lc 'find /target -mindepth 1 -delete && tar xzf /backup/backend-data-YYYY-MM-DD-HHMMSS.tgz -C /target'
```

恢复上传文件卷：

```bash
docker run --rm -v aerolink-prod_backend-uploads:/target -v /home/ubuntu/backups:/backup alpine sh -lc 'find /target -mindepth 1 -delete && tar xzf /backup/backend-uploads-YYYY-MM-DD-HHMMSS.tgz -C /target'
```

恢复完成后重新启动：

```bash
docker compose --env-file .env.production -f "$COMPOSE_FILE" up -d
```

## SQLite 兼容库升级到当前 schema

如果服务器上还是历史 SQLite 结构，先复制一份旧 `prod.db`，对副本运行后端脚本 `npm run db:prepare:legacy-sqlite`，再执行 `prisma db push`。该脚本会先补齐过渡列，再调用 [deploy/sqlite-legacy-backfill.sql](../deploy/sqlite-legacy-backfill.sql) 做字段回填，避免以下信息在收敛 schema 时丢失：

- `customers.address` -> `customers.registeredAddress`
- `quotations.deliveryTerms` -> `quotations.incoterm`
- `quotations.paymentTerms` -> `quotations.commonNote`
- `inventory.status` -> `inventory.conditionCode`
- `inventory.certificateStatus` -> `inventory.certificateType`

当前云端热修就是按这套规则处理的，后续再次重建 SQLite 兼容环境时可以直接复用。

## PostgreSQL 正式迁移建议

从当前云端 SQLite 兼容态切回 PostgreSQL 时，建议按下面顺序执行：

1. 先做代码与数据库双备份。
2. 在 SQLite 副本上执行 `npm run db:prepare:legacy-sqlite`，再用当前 schema 校平数据形态。
3. 在独立环境拉起 `docker-compose.prod.yml`，先使用一次性 `db push` 准备空库并导入清洗后的数据，再完成迁移 baseline、`migrate deploy` 重复演练和恢复验证；不要把 `ALLOW_DB_PUSH=true` 留在正式运行环境。
4. 验证通过后再安排短停机窗口切流，不要直接在现网容器里原地切 provider；正式入口必须设置 `SCHEMA_MIGRATION_MODE=migrate` 与 `MIGRATION_BASELINE_CONFIRMED=true`。

正式执行请使用 [PostgreSQL正式切库执行清单.md](PostgreSQL%E6%AD%A3%E5%BC%8F%E5%88%87%E5%BA%93%E6%89%A7%E8%A1%8C%E6%B8%85%E5%8D%95.md)，其中包含停机窗口、备份、导入、验收和回滚步骤。

## 依赖安全审计

当前环境如果直接执行 `npm audit`，可能会因为镜像站不支持审计接口而失败。请在仓库根目录使用下面的脚本，对前端和后端全量依赖执行审计，并强制走官方 npm registry：

```bash
npm run audit:deps
```

如果锁文件更新后要把修复部署到线上，执行“发布更新”里的前端更新流程即可。
