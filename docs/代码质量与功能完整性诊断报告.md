## 一、总体评价

项目处在 **"MVP 已上线、核心链路扎实、外围模块未落地"** 的阶段。

核心成交链路（RFQ → 询价 → 报价审批 → 发送 → 客户确认 → 自动建单 → 合同生成）从前端到后端到数据库到 E2E 测试已完整闭环。`docs/功能补全计划.md` 记录了 49 条已修复项（F-1 ~ F-49），产品已部署至 `101.43.50.30:8080`。

但 **4 个外围业务模块依然是纯 Mock UI**，没有任何后端对接，与 README 所宣称的"完整功能"存在明显落差。

**一句话定性：主干能卖，枝叶是展板。**

## 三、代码质量

### 3.1 做得扎实的部分

- **后端类型安全**：25 个 route 全部移除 `any`，使用 `Prisma.*WhereInput` + `AuthRequest`（F-6）。
- **分层清晰**：`routes / middleware / lib(service)` 分离，`middleware` 有 `auth / errorHandler / rbac / validate / webhookAudit` 五件套。
- **安全基线完整**：JWT access + refresh、bcrypt、AES-256-GCM 加密邮箱 authCode、Helmet、限流、文件魔数校验、Puppeteer 池化，均已落地。
- **构建配置合理**：`vite.config.ts` 配置了 6 组 `manualChunks`（`vendor-radix / vendor-charts / vendor-forms / vendor-table / vendor-icons / vendor-date`），19 个页面 `React.lazy + Suspense`，最大 chunk `vendor-charts` 432 kB（gzip 117 kB），可接受。
- **观测能力**：Pino 结构化日志、前端首屏 / 切页耗时埋点（F-19）、WebhookAuditLog 独立表。
- **Webhook 生态完整**：inbound + outbound + replay + DLQ + audit 齐全，相关文档多达 6 份。

## 四、功能完整性

### 4.1 真正完整的模块

前端 + 后端路由 + Prisma 模型 + 测试四要素齐全：

- 需求归集（RFQ）
- 智能寻源（Sourcing + AgentWorkbench）
- 报价管理（含审批、发送、撤回、客户确认、自动成单）
- 订单管理
- 库存中心（自有/在途/虚拟）
- 客户管理
- 供应商管理（含 Portal、跟进审计时间线）
- Webhook（inbound + outbound + replay + DLQ）
- 合同模板
- 系统设置
- IngestionHub（邮件归集）

### 4.2 **纯 Mock UI 模块**（本次诊断的主要发现）

这 4 个模块能点开、能看到数据、能交互，但 **任何编辑都只在前端内存里，刷新即丢**。

| 模块 | 前端入口 | 引用的 Mock | 后端路由 | Prisma 模型 |
|---|---|---|---|---|
| **OrderTracking** 物流追踪 | `sections/OrderTracking/index.tsx:28` | `mockShipmentTracking / mockCustomsRisks / mockOrders` | ✅ 已实现 `/api/shipment-tracking` | ✅ `ShipmentTracking` / `TrackingEvent` 已定义并接入路由 |
| **ExchangeVMI** 换件与 VMI | `sections/ExchangeVMI/index.tsx:34` | `mockExchangeQuotes / mockVMIAgreements / mockVMIRestockSuggestions` | ✅ 已实现 `/api/exchange-vmi` | ✅ 已补充 `ExchangeQuote` / `RestockSuggestion` 模型并接入路由 |
| **PricingBI** 定价情报 | `sections/PricingBI/index.tsx:30` | `mockMarketIntelligence / mockLostOrderAnalysis / mockInventory` | ✅ 已实现 `/api/pricing-bi` | ⚠️ 后端返回静态/计算数据，仍为 Beta |
| **TechnicalKit** 技术资料 | `sections/TechnicalKit/index.tsx:29` | `mockIPCData` | ✅ 已实现 `/api/ipc` | ✅ `IPCData` 已定义并接入路由 |

**落地成本估算（从低到高）**：

1. **TechnicalKit** ✅ 已完成：`GET /api/ipc` + seed 数据已落地。
2. **OrderTracking** ✅ 已完成：`ShipmentTracking / TrackingEvent` 路由和 seed 数据已落地。
3. **ExchangeVMI** ✅ 已完成：已补充 `ExchangeQuote` / `RestockSuggestion` 模型并接入路由。
4. **PricingBI** ⚠️ 后端路由已落地，返回静态/计算数据，情报数据源打通仍为未来工作。

### 4.3 混用 Mock 的核心模块

- `src/lib/agentOrchestrator.ts:16` 同时 `import` 真实 `customerApi / supplierApi / rfqApi` 和 `mockCustomers / mockSuppliers`。AGENT 主链路已走真实 API（F-20 起），但本地仍保留 Mock 作为 demo 兜底。**需确认生产环境是否已屏蔽这条 fallback**。
- `src/lib/supplierCapability.ts:1` 引用 `mockSupplierPortalUsers`。该文件参与智能寻源的供应商能力评分，Mock 数据意味着评分基准不真。

## 八、结论

- **核心成交链路（RFQ → 报价 → 订单 → 合同）工程质量高，可投入生产使用**。
- **外围 4 个模块（OrderTracking / ExchangeVMI / PricingBI / TechnicalKit）需要做明确的产品决策**：落地、裁剪或延期，当前"展板状态"应避免继续对外宣传为已交付。
- **建议立即处理的 3 项**：清理 lint、给 4 个 Mock 模块打 Beta 标签、SQLite 迁移计划排期。
