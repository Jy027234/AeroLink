# H0-03 测试环境可验证发布流程

本流程只适用于已获批准的 AeroLink 云端**测试环境**发布。它不授权域名、DNS、TLS、公开生产切流或直接在云端长期修改源码。

`deploy/cloud-release.sh` 现在把以下内容作为不可绕过的发布门禁：

- `SOURCE_REF` 和 `RELEASE_TAG` 必须是相同的、40 位小写 Git SHA；
- 该 SHA 必须与同一轮 Build and Verify 产出的 `release-manifest.env` 一致；
- 必须保存 `backend.sbom.spdx.json` 和 `frontend.sbom.spdx.json`；
- 发布前必须能创建并校验 PostgreSQL 备份；
- 发布后必须通过健康检查、`prisma migrate status`、migration 数量核对，以及库存、金额、交易状态、JSON 配置四类对账；
- 任一发布后验证失败时，脚本会尝试恢复上一对 backend/web 镜像，并把结果记录到 release record。

## 发布前置条件

1. 使用一个已推送的完整提交。先在本地仓库确认它就是远端 `main`：

   ```bash
   SOURCE_REF="$(git rev-parse HEAD)"
   git ls-remote origin refs/heads/main
   ```

   返回的 SHA 必须与 `SOURCE_REF` 完全一致。

2. GitHub 上该 SHA 的 CI 和 Build and Verify 均成功。下载该 Build and Verify 的
   `aerolink-sbom-<SHA>` artifact；解压后的目录必须包含：

   ```text
   backend.sbom.spdx.json
   frontend.sbom.spdx.json
   release-manifest.env
   ```

   `release-manifest.env` 由 workflow 生成，其中的 `SOURCE_REF` 与 `RELEASE_TAG` 必须都是本次 SHA。旧 artifact 若没有此文件，不能用于此流程。

3. 目标必须是独立的 PostgreSQL 测试环境，且已经完成备份/演练库的 migration baseline。历史开发容器若没有 `_prisma_migrations`，不能作为此步骤的数据库或验收证据。

4. 发布目录必须有已审批的 `.env.production`、Docker secret 和正在运行的 PostgreSQL 服务；不得把这些机密或数据库备份复制回仓库。

## 从受控源码构建测试镜像

云端部署目录没有 `.git` 时，先从已确认的 SHA 生成源码归档，并经现有审批渠道传送到测试主机。不要直接在测试主机编辑源码。

```bash
git archive --format=tar "$SOURCE_REF" | gzip > "aerolink-$SOURCE_REF.tar.gz"
```

在测试主机将该归档解压到预先备份过的发布目录后，以完整 SHA 作为镜像 tag 构建：

```bash
export SOURCE_REF=<40 位已推送 SHA>
export RELEASE_TAG="$SOURCE_REF"
export BACKEND_IMAGE=aerolink-prod-backend
export WEB_IMAGE=aerolink-prod-web
export SBOM_DIR=/secure/path/aerolink-sbom-$SOURCE_REF

docker build --file deploy/Dockerfile.server \
  --tag "$BACKEND_IMAGE:$SOURCE_REF" .
docker build --file deploy/Dockerfile.web \
  --build-arg VITE_API_URL=/api \
  --tag "$WEB_IMAGE:$SOURCE_REF" .
```

对全新 PostgreSQL 测试库，必须先按照 migration baseline 演练完成初始化；不要借由设置 `ALLOW_DB_PUSH=true` 绕过正式迁移流程。`MIGRATION_BASELINE_CONFIRMED=true` 仅能在该演练、备份恢复和 `migrate deploy` 均得到验证后设置。

## 执行与留存证据

在已获批准的测试发布窗口内执行：

```bash
SOURCE_REF="$SOURCE_REF" \
RELEASE_TAG="$SOURCE_REF" \
SBOM_DIR="$SBOM_DIR" \
bash deploy/cloud-release.sh
```

脚本会输出 release record 的路径。该 record 应保留并受限访问；其中关联了完整 SHA、镜像 ID、来源 manifest、SBOM 副本、数据库备份、回滚镜像和验证结果。当前源码应核对 **11 条** Prisma migration；脚本会按部署源码目录动态计算该数量，并在 record 中写入 `VERIFIED_MIGRATION_COUNT` 与 `VERIFIED_AT`。

发布脚本自动执行 `deploy/cloud-verify.sh`。如需在不修改环境的前提下重复验收，可使用：

```bash
RELEASE_RECORD=/secure/release-record.env \
SOURCE_REF=<40 位已推送 SHA> \
bash deploy/cloud-verify.sh
```

## 回滚抽查

回滚会切换到 record 中保存的上一对 backend/web 镜像，并等待健康检查。它属于有状态环境操作，只能在获批准的测试窗口内执行：

```bash
RELEASE_RECORD=/secure/release-record.env \
bash deploy/cloud-rollback.sh
```

发布脚本的自动回滚仅在新版本健康检查或发布后验证失败时触发。它不会自动恢复数据库备份；数据库恢复必须依据 release record 中的 `DB_BACKUP`，在人工确认后执行。回滚后应重新运行 `cloud-verify.sh` 或记录明确的回滚验收结果。
