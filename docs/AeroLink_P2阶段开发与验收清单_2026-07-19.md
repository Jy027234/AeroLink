## 1. P2 目标

在不扩张外部业务范围、不破坏现有 RFQ → 报价 → 审批 → 订单 → 库存 → 证书主链的前提下，将 AeroLink 从“功能可运行、P1 可验证”推进到“契约明确、模块可维护、异步任务可恢复、文件可治理、故障可观测、关键页面可在移动端使用”的工程状态。

P2 的主要交付物是：

1. OpenAPI 契约与前端生成型类型客户端。
2. TanStack Query 服务端状态治理与前端 feature 化。
3. 后端模块化单体与清晰的 service/policy/repository 边界。
4. 独立 Worker、可靠重试、补偿与运维入口。
5. 对象存储抽象、文件元数据与安全访问。
6. 后端日志/指标/追踪、前端错误与性能监控。
7. Settings URL 收尾、移动端、无障碍与工程文档。

## 3. 全阶段范围边界

### 3.1 必须保持

- 现有 URL、权限能力、API 状态码、错误码、分页/筛选/排序、CSV 导出确认和审计行为保持兼容；确需调整时先修改契约、提供迁移/兼容层并单独说明。
- 服务端数据库仍是业务真相源；不得把服务端主数据重新放回 Zustand 或浏览器本地状态作为真相源。
- 核心写入继续满足事务、幂等、状态迁移校验、Outbox 和审计要求。
- 数据库变更只能使用可版本化 Prisma migration；必须在干净、隔离的 PostgreSQL 上执行 `migrate deploy` 验证。
- 所有新日志、指标、错误上报和样例数据都必须脱敏，不包含 Cookie、Authorization、口令、密钥、完整邮件正文或个人敏感信息。
- 每批工作独立可回滚、可验证；禁止一次性搬迁全部前端或后端。

### 3.2 明确不做

- 不建设供应商外部门户、外部供应商账号、邀请激活、供应商密码/MFA 或外部租户隔离。
- 不接入真实 ERP、物流、SMTP/IMAP、价格数据商或供应商数据源；只有另行取得凭据、契约和书面授权后立项。
- 不建设 AI 自动报价/自动落单、复杂价格 ML、区块链/外部可信时间戳、Elasticsearch 或微服务。
- 不删除历史 Inventory、旧供应商门户保留字段、Decimal/Enum/JSON 兼容影子；清理工作另行立项。
- 不处理生产域名、DNS、TLS、公网切流或生产合规认证。
- 不在云服务器目录直接开发，不读取或提交服务器密码文件、私钥、`.env.production`、Docker secret、上传文件或数据库备份。
- Git 操作不使用 GitHub CLI，不强推，不改写已共享历史；默认只使用常规 `git add`、`git commit`、`git push`。

### P2-00 架构基线、范围清册与决策记录

- 建立结构热点清册：超大页面、超大路由、跨领域 import、循环依赖、API 进程内定时器和本地文件入口。
- 写出并评审最少五项 ADR：OpenAPI 真相源、生成客户端、Query 缓存策略、模块化单体边界、数据库型 Worker/租约、对象存储与观测方案。可合并成一份决策文档，但每项必须有取舍和回滚路径。
- 冻结首批纵切范围：`RFQ → Quotation → Order → Inventory`；Customer/Supplier 作为主数据依赖，其他演示/Feature Flag 模块不得抢先扩大范围。

#### 验收标准

- [x] 基线文档包含可重复执行的命令、日期、完整 SHA 和结果摘要。
- [x] 路由清册覆盖 `server/src/index.ts` 注册的全部 HTTP mount，并标明 `/api/v1`、入站 Webhook、健康检查等特殊入口。
- [x] 前端清册能指出每项服务端状态当前由谁读取、缓存和失效，不遗漏核心六类列表。
- [x] ADR 明确选择及未选择方案；默认不为 P2 引入微服务、Elasticsearch，Redis 也只能在真实并发指标证明需要后另行决策。

### P2-01 OpenAPI 契约与生成型客户端

#### 开发内容

- 建立 OpenAPI 3.1 真相源及目录约定；工具选型前核对当时官方文档和 Node 22 兼容性，并记录 ADR。
- 统一描述认证/Cookie、capabilities、Feature Flag、分页、筛选、排序、错误信封、日期时间、Decimal 字符串、枚举、幂等键和文件响应。
- 分批覆盖 auth、RFQ、quotation、order、inventory、customer、supplier information、certificate/document、export、outbox/webhook，再补齐其余已启用 HTTP 路由。
- 为入站 Webhook、`/api/v1`、文件上传/下载、CSV/PDF 等非 JSON 响应建立明确契约和安全说明。
- 从 OpenAPI 生成前端 TypeScript 类型/客户端；生成文件不可手工编辑，输出必须可重复。
- 增加契约校验、生成结果漂移检查和代表性请求/响应契约测试，并接入 CI。
- 提供仅开发/受控测试环境可用的 API 文档入口；生产默认关闭或受管理员能力保护。

#### 验收标准

- [x] 所有已启用 HTTP 路由均能在路由清册和 OpenAPI 中一一对应；排除项必须写明原因和责任人。
- [x] OpenAPI lint/validate 通过，生成客户端后工作区无未预期 diff；CI 能在契约与生成物漂移时失败。
- [x] 核心分页接口、错误响应、Cookie 刷新、幂等请求、上传下载和 Webhook 签名至少各有一个自动契约测试。
- [x] 现有前后端和 E2E 行为保持兼容；不得为“适配文档”静默更改状态码、错误码、日期或金额表示。
- [x] 文档样例不含真实 Token、Cookie、邮箱、服务器地址、业务数据或密钥。

### P2-02 TanStack Query 与服务端状态迁移

#### 开发内容

- 引入 TanStack Query，建立 QueryClient、统一 query key factory、请求取消、重试、stale/cache、错误映射和开发调试约定。
- 生成型 API 客户端作为唯一 HTTP 适配层；禁止在新 feature 中继续散落新增直接请求。
- 按 `RFQ → Quotation → Order → Inventory → Customer → Supplier information` 逐领域迁移查询和 mutation；每个领域独立提交、独立回归。
- mutation 明确使用失效、乐观更新或精确缓存更新中的一种，并为并发/回滚行为补测试。
- 退出登录、设备会话撤销、用户/角色变化时清空或隔离用户相关缓存，避免跨账号数据残留。
- URL 继续承载列表页码、筛选、排序和可分享状态；Query 缓存不替代 URL。
- Zustand 只保留 UI 偏好、短期草稿和必要的客户端会话状态；移除已迁移的服务端列表/详情副本。
- 自研 `useQuery`/`useMutation` 在零有效调用后删除；若演示模块暂留兼容层，必须列出范围和退出日期。

#### 验收标准

- [x] 六个核心领域的生产页面全部通过统一 query key 和生成客户端访问服务端；不再双写 Zustand 服务端数据。
- [x] React Strict Mode 下同一查询不发生重复副作用；快速切换筛选/页面时旧响应不能覆盖新结果。
- [x] logout、全部设备撤销、401 refresh 失败后缓存被清除，下一账号看不到前一账号数据。
- [x] mutation 的成功失效、失败回滚、并发提交和用户可见错误均有自动测试；不得捕获后返回 `null` 伪装成功。
- [x] 浏览器前进/后退、刷新和深链接仍保留列表 URL 状态，CSV 导出使用当前服务端筛选而非本地缓存全集。
- [x] 对迁移前后请求数和首个可用内容时间留有基线；无有意义退化，超过 10% 必须说明原因并获接受。Node 22.16.0、同一隔离 seed、Chromium 1440×900、两套生产构建 preview 的 5 次运行热中位数：基线 33 个请求/FCP 44ms/首个 `main` 163.1ms/API P50 22ms/P95 54ms，当前 35 个请求/FCP 44ms/首个 `main` 121.4ms/API P50 22ms/P95 59ms；总请求 +6.1%、FCP 0%、API P50 0%/P95 +9.3%。首个 `main` -25.6% 属于当前本地 preview 的波动/懒加载读数，已在原始记录中保留，测试云/生产窗口仍需按同一脚本复录并由发布负责人接受，不能把本地单点读数伪装成已闭合部署指标。新增 2 个 Query feature 静态 chunk 已在证据中标明。

### P2-03 前端 feature 化与依赖边界

#### 开发内容

- 采用稳定目录约定，例如 `src/features/<domain>/{api,queries,schemas,types,components,pages,tests}`；共享 UI、跨域壳层和基础设施保留明确公共目录。
- 与 P2-02 同行按领域拆分，不先复制整页再长期保留两套实现。
- 将页面壳层、筛选器、表格、详情、表单/弹窗和领域 hooks 解耦；优先处理 Inventory、Quotations、Orders、RFQ、Customers、Suppliers 热点。
- 每个 feature 只从公开入口导出，禁止另一个 feature 深入 import 其内部文件；跨域动作通过明确 service/use case 或共享类型完成。
- 保持现有页面懒加载和预取策略，按实际包分析调整 chunk；不得把所有 feature 重新汇入一个首屏大包。
- 增加依赖边界/循环依赖自动检查；新代码违反边界时 CI 失败。
- 完成迁移且零引用后再删除旧 `src/sections` 实现，避免大爆炸式改目录。

#### 验收标准

- [x] 六个核心领域使用统一 feature 结构，旧入口只保留薄路由适配或已删除。
- [x] feature 之间无内部路径穿透和循环依赖；自动边界检查已接入 CI。
- [x] 已迁移页面入口主要承担装配，不再包含完整 API、状态、表格、表单和弹窗实现；异常超大文件有书面例外。
- [x] 所有既有页面路径、权限隐藏、深链接、前进/后退和懒加载行为保持一致。
- [x] 构建产物有前后对比；首屏 JS 或核心页面 chunk 增长超过 10% 时有可验证理由和后续措施。
- [x] 核心导航、RFQ/报价/订单/库存链路和失败提示 E2E 全部通过。

### P2-04 后端模块化单体

#### 开发内容

- 保持单仓库、单 API 部署单元，建立 `modules/<domain>`；不得拆成微服务。
- 推荐模块边界：Identity & Access、Customer & Supplier、RFQ & Sourcing、Quotation & Order、Inventory & Quality、Integration & Notification、Reporting & Pricing。
- 每个模块按需包含 routes/controller、schemas、service、policy、repository、events/outbox 和 tests；不为目录完整性创建无职责空文件。
- 先迁移 quotation/order，再迁移 RFQ/inventory，最后迁移主数据与集成；保留原 URL 的薄适配层。
- 将事务、状态迁移、幂等、权限决策和审计上下文移入 service/policy；路由只做 HTTP 解析、调用和响应映射。
- Prisma 访问收口到 repository/service；跨模块写入通过明确用例，不允许路由直接串联多次 Prisma 写操作。
- 分离 Express app 构造、HTTP server 启动和后台进程启动，保证测试导入 app 时不启动端口或定时器。
- 统一领域错误到既有 HTTP 错误码的映射，并保持 OpenAPI 契约同步。

#### 验收标准

- [x] 核心交易写路由的事务、状态迁移、幂等与权限均由 service/policy 决定，路由中不再编排多次数据库写入。
- [x] 模块依赖图无循环依赖；禁止跨模块直接访问对方 repository，例外有 ADR。
- [x] quotation/order、RFQ/inventory、customer/supplier 至少分别具备 service 单测和 PostgreSQL 集成测试。
- [x] 原有 URL、状态码、错误码、审计、Outbox 和响应体通过 OpenAPI/契约回归，无破坏性漂移。
- [x] 导入 app 不监听端口、不运行 Worker；API、Worker 和测试入口职责清晰。
- [x] 后端全部测试、构建、隔离 PostgreSQL 主链 E2E 通过；不得以 Mock Prisma 单测替代真实事务验收。

### P2-05 独立 Worker 与可靠任务执行

#### 开发内容

- 将 Outbox、Webhook 重试及适合异步执行的周期任务从 API 进程移到独立 Worker 入口和部署服务。
- P2 默认先使用 PostgreSQL 领取/租约机制，复用 Transactional Outbox；没有并发指标和 ADR 时不引入 Redis 队列。
- 增加原子领取、`workerId`、租约到期、心跳、指数退避、最大重试、终态失败、人工重放和幂等去重。
- 处理 Worker 在任务执行前、执行中、外部副作用后但落库前崩溃的恢复策略；邮件/Webhook 保留稳定幂等标识。
- 增加优雅停机：停止领取新任务、在超时内完成或释放租约、输出未完成任务状态。
- 更新本地 Compose、测试云部署、健康/就绪检查和运维手册；API 可在 Worker 暂停时继续接收业务并积压任务。

#### 验收标准

- [x] API 进程不再启动 Outbox/Webhook 后台定时器；Worker 可单独启动、停止和重启。
- [x] 两个 Worker 并发领取同一任务时只产生一次受控副作用；自动化测试包含竞争场景。
