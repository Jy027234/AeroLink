### 外部 collector 留存证据格式（部署后提供）

新增 `scripts/verify-observability-retention-evidence.mjs`，用于在部署窗口收到外部 collector 的脱敏留存记录后做只读格式与一致性校验。它要求完整 release SHA、采集时间、collector origin、metrics/trace 留存策略，以及两类信号各至少一个已删除探针；会拒绝 URL 凭据、query/fragment、敏感字段和 credential-like 值，并只输出 origin、策略和探针年龄等摘要。该工具验证的是**证据格式与声明内部一致性**，不连接 collector，也不把本地 `retention-check` 结果当作外部删除证据。

部署后保存一份不含凭据的 JSON（不要把真实 token、邮件正文、payload 或对象内容提交到仓库），按下列形状生成并校验：

```json
{
  "schemaVersion": 1,
  "environment": "<test-cloud|production>",
  "releaseSha": "<40-char-full-sha>",
  "capturedAt": "<ISO-8601>",
  "collector": { "name": "<collector-name>", "endpoint": "https://<collector-origin>" },
  "policy": { "metricsRetentionDays": 7, "traceRetentionHours": 24 },
  "checks": [
    { "signal": "metrics", "probeHash": "<sha256>", "createdAt": "<ISO-8601>", "queriedAt": "<ISO-8601>", "observed": "deleted" },
    { "signal": "traces", "probeHash": "<sha256>", "createdAt": "<ISO-8601>", "queriedAt": "<ISO-8601>", "observed": "deleted" }
  ],
  "source": { "kind": "deployment-verification", "reference": "<release-record-or-ticket>" }
}
```

PowerShell（直接使用 Node 22，避免 shell 对 npm 透传参数的差异）：

```powershell

& "$nodeRoot\node.exe" scripts/verify-observability-retention-evidence.mjs --evidence=<脱敏证据 JSON 路径>
```

## 内部试点 SLO、留存与故障处置

以下阈值与当前内存指标/告警实现保持一致；它们是内部测试云和本地验收门槛，不代表公网生产承诺：

| 信号 | 目标 | 告警条件 | 确认入口 |
|---|---|---|---|
| API 5xx | 5xx < 0.5%（样本至少 10） | `api.error-rate`：错误率 ≥ 5%；≥ 20% 为 critical | `GET /api/metrics`、request/trace 日志 |
| API 延迟 | P95 < 1,000 ms | `api.latency-p95`：P95 ≥ 1,000 ms；≥ 3,000 ms 为 critical | `GET /api/metrics` |
| 数据库延迟 | P95 < 1,000 ms | `database.latency-p95`：至少 5 次查询且 P95 ≥ 1,000 ms | `GET /api/metrics` |
| 数据库错误 | 错误率 < 5%（样本至少 10） | `database.error-rate`：错误率 ≥ 5%；≥ 20% 为 critical | `GET /api/metrics`、无 SQL 的错误日志 |
| 异步任务 | 不允许静默丢失 | Outbox/Webhook `FAILED`、重试耗尽或 lease 恢复告警 | 受控 Outbox 查询、Worker 日志、`npm run worker:fault-check` |
| 文件访问 | 未授权访问必须为 401/403 | `object-missing` 或校验失败告警 | 受控文件下载日志、对象存储 contract test |

当前进程内留存是有界的：HTTP/数据库各保留最近 1,000 个样本，trace 保留最近 500 个 span，alert 仅保留当前活动键；进程重启会清空这些聚合数据，不保存请求 payload。若配置 `TRACE_EXPORTER_URL`，只发送脱敏 span，外部系统的留存期限必须在部署记录中明确（内部试点建议 metrics 7 天、trace 24 小时）；未配置采集端时不得把“内存数据”当作长期审计证据。

### 故障处置步骤

1. **确认范围**：保存当前时间窗口、`requestId/traceId`、告警 key 和状态码摘要；不要复制 Authorization、Cookie、邮件正文、完整 webhook payload 或对象内容。
2. **API/数据库变慢**：先暂停非必要批处理和受控导出，保留核心 RFQ→报价→订单写入；检查 `/api/metrics` 的 P50/P95、DB slowQueries 与最近错误日志，必要时回滚最近应用版本。
3. **Worker 重试/积压**：停止领取新任务但保持 API 接收；确认 lease、`workerId` 和 `attemptCount`，使用受控管理入口逐项重放，先验证幂等键和审计记录，再恢复 Worker。不得直接修改状态字段。
4. **对象缺失/校验失败**：阻断该对象下载并保留元数据；核对 SHA/size 与对象存储，优先从已确认备份恢复，禁止绕过逐对象授权或把本地目录直接暴露为静态入口。
5. **恢复验证**：重跑 `npm run observability:check`、相关 reconciliation 和受影响 E2E；确认活动告警已 resolve、request/trace 可关联，再记录恢复时间与回滚决定。
