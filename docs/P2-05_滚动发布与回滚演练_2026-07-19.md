# P2-05 Worker 滚动发布与回滚演练

日期：2026-07-19（Asia/Shanghai）
状态：代码发布合同已通过；真实测试云执行需绑定新的完整 SHA 与发布窗口

## 兼容性决策

P2 新 Worker 使用 `workerId/lockedAt` 租约字段，而基线版本的旧进程不了解租约条件更新。两者不允许同时领取同一批 Outbox 任务；发布方案采用“停止旧 Worker、确认租约可恢复、切换 backend/web、启动新 Worker”的 drain 策略，而不是假设旧/新二进制可以安全并行。

迁移只新增可空字段和索引，旧版本可读取既有业务列；如果旧 Worker 在停止超时后仍留下 `PROCESSING` 任务，新 Worker 会按 stale lease 规则回收，不直接删除或改写业务 payload。

## 执行顺序

1. 校验 `RELEASE_TAG == SOURCE_REF`、SBOM/manifest、数据库备份和 migration 计数。
2. 记录当前 backend、worker、web image ID；停止旧 worker，等待其优雅停机窗口（Compose `stop_grace_period=45s`）。
3. 切换 backend/web 到 release tag，等待 `/api/health` 返回 `status=ok`，由 backend entrypoint 执行受门禁的 migration deploy。
4. 仅在 backend 健康后启动 release worker；`cloud-verify.sh` 同时校验 release record、migration 数量、worker 容器 image ID、retention-check 配置门禁和四类 reconciliation。
5. 观察一轮 Outbox/Webhook lease、attempt、失败告警和核心链路；确认没有旧 worker 容器后才结束发布。

## 回滚顺序

1. 停止当前 worker，保留数据库和 release record/备份。
2. 恢复 backend、worker、web 的 rollback image；不得删除已部署 migration，不使用 `db push` 逆向 schema。
3. 启动 backend/worker/web，等待健康检查和 reconciliation；旧 worker 只在新 worker 已停止后重新加入。
4. 将 release record 标记为 `rolled_back_manual` 或 `rollback_failed`，留存告警、request/trace 和恢复验证结果。

## 回滚边界

- 发布脚本只允许完整 40 位 SHA，禁止把可变分支名写入 release record。
- worker 与 backend 默认共用同一镜像，但分别记录 rollback/release ref，支持未来拆分镜像。
- 旧/新任务 schema 若未来发生不兼容，必须先停领并排空旧任务，或增加显式版本化 eventType/consumer；不得在滚动发布中静默混用。
