# P1-01 交易状态治理迁移说明

迁移 `20260715143000_transaction_status_governance` 为 RFQ、报价单和订单增加 `version` 乐观锁字段，并新建追加式的 `transaction_status_history` 审计表。每条历史记录保留实体类别、前后状态、原因码、可选原因、操作者、状态版本和发生时间。

## 上线前检查

1. 先备份 PostgreSQL 数据库，并确认应用实例已停止处理状态变更请求。
2. 以只读账号执行以下检查，确认 RFQ 仅含可识别状态或记录待人工处理的异常值：

```sql
SELECT "status", COUNT(*)
FROM "rfqs"
GROUP BY "status"
ORDER BY "status";
```

3. 使用应用 Node 22 运行 `npx prisma migrate deploy --schema prisma/schema.prisma`。迁移会将旧 RFQ UI 别名规范化为状态机内部值：`APPROVED → APPROVING`、`SENT → ORDERED`、`WON → COMPLETED`、`LOST → CANCELLED`。

## 数据回填

迁移在历史表中为已有 RFQ、报价和订单各写入一条 `MIGRATED_INITIAL_STATE` 记录，使用原业务记录的创建时间与可用创建者。它表示 P1-01 上线前可恢复到的最早状态快照；不会伪造无法追溯的历史迁移链。

新建业务记录和之后的每次真实状态变更由应用在同一数据库事务内写入历史表。状态更新使用 `id + 当前状态 + version` 条件更新；条件不匹配时 API 返回 `409 STATE_CONFLICT`，客户端应刷新后再提交。

## 回滚边界

Prisma 不执行自动 down migration。仅在迁移刚完成、且没有任何 P1-01 应用实例写入新的状态历史时，才可在已验证备份可用的情况下执行下列回滚 SQL：

```sql
BEGIN;
DROP TABLE IF EXISTS "transaction_status_history";
ALTER TABLE "orders" DROP COLUMN IF EXISTS "version";
ALTER TABLE "quotations" DROP COLUMN IF EXISTS "version";
ALTER TABLE "rfqs" DROP COLUMN IF EXISTS "version";
COMMIT;
```

若已有新的状态变更，禁止直接删除历史表或版本列。应先停止写流量、备份审计表，再通过修复发布或由数据库负责人制定保留审计数据的补偿方案。RFQ 别名规范化是数据修复步骤，回滚时不应盲目将状态改回旧别名。
