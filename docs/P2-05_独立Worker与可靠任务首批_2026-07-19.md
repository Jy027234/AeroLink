# P2-05 独立 Worker 与可靠任务首批

日期：2026-07-19（Asia/Shanghai）
状态：首批代码完成（API/Worker 生命周期、数据库租约、隔离故障注入、三处真实进程终止、人工重放安全门禁和 drain 发布合同已通过；真实测试云发布仍受发布授权约束）

## 已交付

- 新增 `server/src/worker.ts` 工厂和 `server/src/worker-entry.ts` 独立进程入口；`npm --prefix server run worker` 可单独启动。
- API 默认不再创建 Outbox/Webhook/幂等清理定时器；仅受控 E2E fixture 通过 `ENABLE_INLINE_WORKER=true` 临时启用内嵌 Worker。
- `docker-compose.yml` 开发服务与 `docker-compose.prod.yml` 已增加独立 worker 服务；开发启动改为 `prisma migrate deploy`，不再以 `db push` 作为默认 schema 门禁。
- Worker 支持优雅 stop：停止领取新任务、等待进行中的扫描在超时内完成，超时则记录待恢复租约；随后按独立间隔调度 Webhook 重试、Transactional Outbox 和幂等清理。
- Webhook 重试增加 `processing` 原子领取、独立 `lockedAt`、`workerId` 和 5 分钟 stale-lease 恢复；Outbox 保留现有 `updateMany` 领取/锁超时恢复语义，并为长任务增加心跳续租。
- 任务完成/失败/取消均带当前 `workerId` 条件，失去租约的旧 Worker 不能覆盖新领取者的终态；Webhook 外部副作用返回后的成功、失败、异常和 inactive 分支也用 `status + workerId` 条件落库；Webhook lease 迁移为 `20260719130000_p2_webhook_leases`。
- Outbox 在“已原子认领但行随后消失”的异常窗口会条件释放 `PROCESSING` 租约并记录告警，避免留下不可恢复的幽灵锁；该分支有单测覆盖。
- 人工重放必须同时满足 `outbox.manage` 能力和请求体 `confirm=replay`；成功与失败均写入 `REPLAY/OUTBOX` 审计，审计只保留事件 ID、结果和错误摘要，不记录事件 payload。

真实进程终止演练入口已固化为 `npm run worker:crash-check`：在同一个隔离 PostgreSQL 中创建合成 Webhook 任务，分别让独立子进程在 `before-dispatch`、`during-side-effect`、`before-finalize` 三个故障点以退出码 70 终止；父进程将租约置为已过期后启动恢复 Worker，逐项验证任务回收为 `DELIVERED`、`workerId/lockedAt` 释放和 attempt 累加。该入口仅在显式 `P2_FAULT_INJECTION=true` 时可用；本次已在项目 PostgreSQL 容器中新建并清理临时数据库完成三点演练，未触碰真实业务文件。
